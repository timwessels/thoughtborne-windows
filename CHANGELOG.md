- Archived transcripts and recordings now name the engine that actually produced the text (#62). The archive filenames carried only a timestamp, so which engine made a given transcript lived only in `thoughtborne.log` — and only as long as it survived. Each saved transcript now gets a short, self-explanatory engine token appended to its name — `text_<ts>_<code>.txt`, with `code` one of `s-live` (Soniox Live), `s-v2` / `s-v4` (the `soniox` upload slot's V2 sync or V4 async engine), `groq`, or `groq-large` — and the paired recording is renamed to match (`voice_<ts>_<code>.mp3`) after transcription succeeds, the only moment the winning engine (fallback chains included) is actually known: the archived MP3 is written *before* transcription, so any audio-side token must be a post-transcription rename. The token names the engine that *really* produced the saved text: the hybrid `soniox` slot reports its per-recording V2-vs-V4 choice through a per-call sink (thread-safe across the parallel workers, unlike a mutable attribute on the shared transcriber singleton), so even the rare V2-raises-then-V4-rescues path is labelled `s-v4` instead of mislabelled `s-v2`, and the empty-live fallback cascade returns the engine that broke the silence. Everything degrades defensively (stability is principle #1): an empty or unknown code, a missing source, or a locked archive file (e.g. an AV scanner still holding the just-written MP3) leaves the bare `voice_<ts>.mp3` / `text_<ts>.txt` in place with a single warning — the audio↔transcript pairing keys off the shared timestamp, which both files still carry, so a failed rename never breaks the pairing or the `Ctrl+Alt+R` retry. Recordings that were never transcribed (exit/device-loss salvage, kill-recovery) keep their bare stem, since no engine produced anything to name. Deliberately out of scope for v1: no backfill of the existing `history/` archive (tokens start from here, not retroactively from the log), and a retry (`Ctrl+Alt+R`) tokens only the new transcript, not the old audio — the retry already writes a fresh timestamp stem and never re-pairs with the original recording, so its engine still shows in the transcript name. Offline name-logic tests live in the local `_tools/issue62/`.
- The audio capture thread now runs at elevated scheduling priority so recordings stop silently losing words under CPU load (#72). The single capture loop drains the mic through a blocking `stream.read(..., exception_on_overflow=False)`; when the thread — or the Windows audio service — is preempted long enough for PortAudio's input buffer to overflow, the excess samples are discarded silently (the "Audio gap detected" case: words vanish as a skip, not as silence), exactly what the unread quality bar exists to prevent. A 7-Zip experiment isolated *scheduling priority* as the lever — extraction in the foreground caused heavy dropouts, and 7-Zip's "Background" button, which lowers its process priority without reducing its workload, stopped them — with the per-session loop-iteration stats showing the capture-thread-starvation signature (loop gaps >50 ms under load, none when clean). The capture thread now registers with the Windows multimedia scheduler (MMCSS "Pro Audio"), the standard mechanism for glitch-free audio, falling back to a direct thread-priority bump if that service is unavailable; and where the process was started externally below Normal priority — a field-observed handicap (base priority 6 on the maintainer's machine) that deprioritises every thread including the live-transcript receiver — it lifts the whole process one notch above Normal, a no-op on any process already scheduled at or above Normal, so unaffected machines stay untouched. Fully defensive: every failure — a missing DLL, a disabled service, an exotic Windows — degrades to the previous default-priority behaviour and never affects recording, mirroring the push-to-talk and device-loss guards; all via `ctypes`, no new dependency, no admin rights, Windows-guarded. One INFO line at recording-loop start records which mechanism took (and, when a lever is skipped, why), so a hands-on load test reads directly against the per-session loop-iteration stats `stop_recording()` already logs. No new config surface — the design is self-degrading and needs no disable switch, and the tunable values (task name, target class, gate) live as named locals in the helper. Priority mitigates load-induced overflow but cannot conjure samples that never arrive, so the Bluetooth read-latency component of dropouts stays a separate track (#70).
- The Soniox V2 sync path now applies the personal recognition vocabulary too (#73). Short recordings on the `soniox` upload slot (under `SHORT_AUDIO_THRESHOLD`, 58 s) run the legacy V2 gRPC engine, which was the only Soniox path still ignoring `personal_settings.json` — the two v4 engines (Soniox Live and the slot's async fallback) already sent it. V2 now derives a `SpeechContext` from the very same `SONIOX_CONTEXT["terms"]` the v4 engines consume (one `SpeechContextEntry`, `boost` from the new `SONIOX_V2_CONTEXT_BOOST` config constant, default `15.0` to match Soniox's own legacy examples), so proper nouns and domain terms ("Claude Code", "Anthropic", "WSL2", …) are recognized as well there as on every other Soniox path — and both the slot's short-recording branch and the empty-live fallback cascade get it through the shared `_transcribe_v2_sync` primitive, a single wiring point. No second vocabulary source: `personal_settings.json` stays the single origin; the `general` domain/topic block (v4-only, no gRPC `SpeechContext` equivalent) is untouched — only its `terms` cross over. The context is built once per session, lazily after the optional SDK import (nothing soniox-typed on the module-import path, #31) and only from non-empty string terms, so a missing SDK or an absent/empty/malformed vocabulary leaves V2 sync byte-identical to before: non-string entries are dropped and any build failure degrades to no context — a broken `personal_settings.json` must never break transcription. One `Soniox V2 sync context enabled: N terms` log line mirrors the v4 engines. Both README twins, `llms-install.md`, and the `personal_settings.example.json` comment are corrected — they claimed the v2 sync path ignored the vocabulary. Groq still ignores it (its `prompt` is only a weak decoder bias, evaluated and declined).

- Optional push-to-talk mode (#66): tap Left-Ctrl, release, then press-and-hold Left-Ctrl — recording runs while held, releasing inserts the transcript at the cursor. It is built on the same hook-free physical-key-state polling the hotkeys already use (`is_key_pressed`), not a low-level keyboard hook — the hook was removed in early 2026 because Modern Standby silently invalidated it, freezing every hotkey until a force-kill; the ~100 Hz recording-loop thread that now also hosts the gesture detector survives standby, so polling is the only safe path (full reasoning in the local `_research/2026-06_push-to-talk-doppeltap-ctrl/`, spike #63). A mandatory AltGr filter makes the German QWERTZ `@ \ { } [ ] | € ~` keys never false-trigger: AltGr is Ctrl+Alt, so the gesture counts Ctrl as a trigger only when it is *bare* (no Right-Alt, no other key down), and any foreign key disarms the gesture mid-arming — so `Ctrl+C` → `Ctrl+V` and every other Ctrl combo are untouched. The feature is **opt-in and off by default**: while disabled the running app never calls into any of the new code, so the existing Ctrl+Alt+W flow is unchanged for anyone who has not enabled it (the new `push_to_talk` block of `personal_settings.json` carries the master switch plus the trigger key, the insert path — type/clipboard/send/no_insert, defaulting to clipboard like Ctrl+Alt+D with type as the fallback for apps that block paste (#68) — and the three timing thresholds; bad values warn and fall back to the default). When enabled, push-to-talk is inert while a Ctrl+Alt+W recording is active and a push-to-talk-owned recording is stopped only by releasing the trigger, not by A/D/H/Y; the gesture detector is confined to the recording-loop thread and interlocked with the existing recording state through one new loop-thread-only flag — no new lock, and the five hotkey callbacks plus `start_recording`/`stop_recording`/`start_processing_thread` are unchanged, the new path only *calls* them. The gesture logic is a Win32-decoupled state machine in the new `ptt_detector.py`, so it can be unit-tested with synthetic key sequences off-Windows. The elevated-window limit (physical-key reads return nothing under Windows' UI-privilege isolation) is identical to the existing hotkeys, not a regression. New `config.py` push-to-talk defaults; new `ptt_detector.py`; `hotkey_manager.py` gains a raw-VK `is_vk_pressed()` helper and the Left-Ctrl/Right-Alt VK constants alongside the untouched name-keyed `is_key_pressed`.
- Never lose a recording: a four-layer safety net for in-flight audio (#49). Frames used to live only in RAM until a recording was stopped — exiting mid-recording (the real incident behind the issue), a crash/kill/BSOD, or a microphone that died for good meant the dictation was unrecoverable. Four cooperating layers close that surface: **(1) Exit salvage** — the exit hotkey, Ctrl+C, and the newly mapped Ctrl+Break (SIGBREAK → KeyboardInterrupt, previously instant process death) now stop an in-flight recording first and persist it through the normal archive path without transcribing; the console states duration and file path. Idempotent across exit paths via a salvage flag, and bounded: no rescue call can block the exit indefinitely. **(2) Crash-safety sidecar** — during recording, a per-session observer thread mirrors the captured frames into `voice_archive/voice_<start-ts>_r16000c1s16le.partial` (raw PCM; rate/channels ride in the name so a later config change cannot silently mis-decode an old sidecar) every `SIDECAR_FLUSH_SECONDS` (5 s guide value, flush + fsync per batch), capping the loss window of an uncatchable death at roughly one batch. The capture hot path is untouched by construction — `record_chunk()` gains no call, no flag, no queue; the writer observes the frames list by index from its own thread (GIL-consistent `len`/slice on an append-only list), so disk stalls block only the writer while frames keep growing in RAM exactly as before. Any sidecar I/O error degrades to today's RAM-only behavior with exactly one warning — a rescue path must never endanger the recording it rescues. The sidecar is pure crash insurance: every in-process path (stop, cancel, exit salvage, device-loss endgame) saves from RAM via `save_recording()` byte-identically to before and only deletes the sidecar — on the normal stop path not before the archive MP3 exists, so even a crash in the stop-to-save window stays covered. A one-byte `msvcrt` lock on the open sidecar keeps a second instance's startup recovery (BAT double-click; the hotkey-less second process still runs its recovery) from stealing a file that is still being written; the OS releases the lock on process death, so crash + immediate restart recovers without waiting. **(3) Startup recovery** — leftover `.partial` files are converted to `voice_<start-ts>_recovered.mp3` during startup, before hotkeys exist; the startup block lists each recovered file and the newest one arms the `Ctrl+Alt+R` retry slot (#24 machinery), so one keypress transcribes the recovered audio. Recovered files carry the recording START time (a crash has no stop time) and are silence-padded but not tail-trimmed (the trim exists to remove the stop-hotkey click — a crash had none; trimming would discard real dictation). Sidecars that fail conversion are renamed to `.partial-unrecoverable` (bytes kept for manual rescue) instead of error-spamming every future start; torn final samples from a mid-write kill are trimmed to sample alignment. **(4) Device-loss endgame** — when the stream-error path gives up after `max_stream_errors` and a failed reinitialize, the recording now ends cleanly instead of zombieing (see Fixed): the captured frames are saved through the normal archive path, the retry slot is armed, and the console says plainly what happened and that reconnecting the microphone plus `Ctrl+Alt+R` transcribes what was saved. New `SIDECAR_FLUSH_SECONDS` config constant; new `_tools/issue49/` test scripts (local) covering byte-equivalence, writer mechanics including the cross-process lock probe, and the recovery degradation paths.

### Changed

- The README Soniox pricing note (both twins) is refreshed against July 2026 pricing (#147): the stale "billed from the first minute" qualifier is dropped — Soniox now bills token-/duration-based on actual audio length, with no per-minute minimum documented anymore — and the async rate is added beside the real-time one ($0.12 per hour of audio for the real-time default, Soniox Live; $0.10 for async file uploads, Soniox). The ~$3/month figure (≈ 25 h × $0.12) is unchanged and re-verified; the Soniox date stamp moves June → July 2026 to match the re-verification, while the un-rechecked Groq free-tier limits keep their June 2026 stamp.
- The README shortcut recipe (both twins) now recommends `cmd.exe /c "…\Thoughtborne.bat"` as the shortcut target (#139). Start-menu entries whose shortcut points straight at a `.bat` never get *Run as administrator* in their context menu — Windows curates that menu and drops the verb for script targets — so admin-window dictation needed a detour through Explorer. With the `cmd.exe /c` target the same single shortcut starts normally on left-click and offers ad-hoc elevation on right-click; click-verified on Windows 11 25H2. The troubleshooting entry now also names what the UAC prompt shows (*Windows Command Processor* — batch files run through `cmd.exe`).
- The startup RECOVERED panel now appears only once per failure — on the next start after a recording was saved but not transcribed — instead of on every start (#134). #114 made the untranscribed-recording marker persistent so `Ctrl+Alt+R` survives any number of restarts, and #106's earlier announce-once UX was the price paid: the panel re-appeared on every start until the recording was retried or superseded (field evidence: a 0.3 s accidental recording drew the panel on all 20 starts over 2.5 days). The marker lifecycle stays exactly as persistent — the recording remains retryable across restarts until a successful retry or a newer failure supersedes it — but the *announcement* is one-shot again: the first start after a failure shows the panel and arms the slot, every later start arms the `Ctrl+Alt+R` slot silently from the still-present marker, and a deliberate `Ctrl+Alt+R` still reaches the recording (declining to recover is a valid choice, not something the tool nags about). A new failure — including a failed retry that hits a transport/API error — resets the announcement to exactly one more panel on the next start. The single "already announced" bit rides in the marker name (`…_seen.needsretry`), so no new file and no second bookkeeping mechanism appear; every pre-existing marker (current or the legacy #106 bare-name form) counts as not-yet-announced and gets exactly one more panel after the update, then goes quiet. The single-slot semantics, the singular panel wording (#126), and the never-delete-audio rule (#114) are all unchanged. Recorded as decision D-001 in the new `DECISIONS.md`, and the marker lifecycle gains an off-Windows regression test, `test_retry_marker_lifecycle.py`.
- `Thoughtborne.bat` now closes its window immediately after a clean exit instead of waiting for a key press (#129). A normal shutdown (`Ctrl+Alt+4`, exit code 0) jumps straight to a new `:done_clean` path and closes; the error and setup branches still pause on "Press any key …" so their message stays readable before the window disappears. `popd`/`endlocal` run on both paths.
- The console-visible Soniox error and fallback wording is de-v4'd to match #121 (#124). The strings #121 left behind — `retrying with Soniox V4 (async REST, slower)`, `Error during Soniox v4 transcription`, `Soniox v4 transcription failed` / `polling timeout`, and the async self-test lines — now derive their name from `get_name()` (`"Soniox async"`), so the console speaks one generation-neutral name. The empty-live fallback cascade's own console error line (`Fallback (...) raised`, `thoughtborne.py`) drops its hardcoded `Soniox V4 (async)` for the same neutral name — a plain literal there, since a failed lazy init can leave no transcriber instance to call `get_name()` on. Internal nicknames, class names, and file-log-only / DEBUG strings are deliberately kept, exactly as in #121.
- Pinned both Soniox models explicitly to v5 and retired the v4-era config names (#121). Soniox removed `stt-rt-v4`/`stt-async-v4` on 2026-06-30 and now silently aliases them to `stt-rt-v5`/`stt-async-v5`, so the tool had been relying on that alias; it now requests the v5 strings directly (`SONIOX_RT_MODEL = "stt-rt-v5"`, and the async pin renamed `SONIOX_V4_MODEL` → `SONIOX_ASYNC_MODEL = "stt-async-v5"`). The whole async-REST config block moves from `SONIOX_V4_*` to generation-neutral `SONIOX_ASYNC_*` names (`_API_BASE`/`_POLL_INTERVAL`/`_MAX_POLL_ATTEMPTS`) with its section headers and comments de-v4'd, so a name can never again contradict the value it pins; the mechanical use-site updates in `transcriber.py` carry no behavior change. The archive engine tokens follow the re-pin, per the #84 contract that the token records the model generation the tool *requested*: future Soniox Live and async recordings are stamped `SonLive-v5` / `Son-v5` instead of `-v4`, while existing archive files keep the token they were written with — no migration, since the audio↔transcript pairing keys off the shared timestamp, not the token, so mixed old/new tokens coexist safely. The async engine's user-visible name is de-v4'd to match (`get_name()` `"Soniox v4 (async)"` → `"Soniox async"`, plus the init-log line and the class/method docstrings), while the internal engine nickname is deliberately kept as a stable code identity distinct from the model pin — the `SonioxV4Transcriber` class, `self._v4`/`kind="v4"`, the `ENGINE_TOKENS` keys, and the deep file-log strings. Both README twins and `llms-install.md` now name the v5 engines (and the v5 archive tokens); `.env.example` and `personal_settings.example.json` drop the stale generation from their Soniox comments.
- The startup RECOVERED panel no longer counts the pending recordings; it always announces a single one (#118). `Ctrl+Alt+R` only ever retries the newest recording (single-slot semantics, #24/#114), so a count like `2 recordings ... -- 1 more in the audio folder` on one restart versus `1 recording` on the next read as an inconsistency in live testing and named something the console user cannot act on in-app. The panel now reads `a recording was saved but not transcribed` (clean-exit salvage) or `a recording was rescued after a hard kill` (kill recovery; both heads gained a `was` in a wording refinement, #126) — the origin distinction stays, derived from the newest pending recording's stem as before — with the origin-neutral `from <when> (<dur>)` detail and the singular `press Ctrl+Alt+R to transcribe & insert it`. Display only: the marker lifecycle, single-slot arming, startup supersede-prune and the never-delete-audio rule (#114) are untouched, and dropping the plural head/detail/what-now branches makes the renderer smaller. `thoughtborne.log` still records the pending count as a debug detail.
- `Ctrl+Alt+R` (retry last failed) now survives any number of restarts (#114). The one-restart limit was an artifact of how each of the three untranscribed-audio sources persisted: an in-session transcription failure (#24) armed the retry slot in memory only, so exiting after it silently lost the offer, while the hard-kill `.partial` recovery and the clean-exit `.needsretry` marker (#106) were each consumed on first read. The `.needsretry` marker is promoted from a one-shot offer to the persistent truth "saved but never successfully transcribed": it is now written at all three sources — keyed on the mp3 *stem* so the kill-recovered `voice_<ts>_recovered(_<n>)?.mp3` shape round-trips alongside the bare `voice_<ts>.mp3`, with the duration still riding in the name (`_d<ms>`) so startup needs no file read — kept across restarts instead of deleted on read, and removed only when the recording is retried successfully or a newer failure supersedes it. Superseded older markers are pruned at both reconciliation points (retry-success and startup), keeping the single-slot "newest wins" semantics of #24, so an already-superseded recording can never resurrect on a later start. Every marker write and delete is best-effort and never-raises (stability #1): the in-memory slot stays the in-session critical path and the marker is a persistence add on top, so a marker fault can never break the recording, exit, or retry path. ### Fixed

- A Soniox Live recording of 58 s or longer whose live transcript came back empty no longer earns the honest "no speech" verdict while the async file engine is unreachable (#141, respects D-001). The async stage of the fallback chain used to swallow all its transport/API errors and report clean-but-empty, and on long recordings it is the *only* file stage (the V2 sync stage is duration-gated off) — so a real outage deleted the retry marker and disarmed `Ctrl+Alt+R` exactly when switching engines would have helped. The async engine now reports its error channels — a job-level `status: "error"` (where the actual `error_message` field is finally read, instead of always logging "Unknown error"), an HTTP transport error, or the poll timeout — through a per-call error sink, so such a recording stays on the conservative FAILED + retryable side, in session and on a manual `Ctrl+Alt+R` retry alike. The same signal also hardens the short-recording path: an outage there (with the V2 sync stage cleanly empty) is likewise treated as FAILED + retryable rather than "no speech", since a chain with a broken stage is not the all-clean run D-001 reserves the honest verdict for. The direct engine slots (`Ctrl+Alt+L`) are unchanged; surfacing their signals is #138. Off-Windows coverage lands in `test_retry_marker_lifecycle.py` — the real fallback chain against a faked-outage V4 stage, plus per-channel tests of the async engine's four error returns.
- A recording that transcribes to empty on every engine no longer nags with a misleading retry offer (#133). The persistent retry marker (#114) is meant to survive restarts for real failures — no network, a broken key — where a later retry can still succeed. But an accidental silent recording (e.g. a 0.3 s push-to-talk tap) runs the whole fallback chain cleanly and comes back empty every time: semantic emptiness is a *final* verdict, yet the failure path treated it the same as a transport error and kept the marker forever. The two outcomes are now distinguished at the point where all fallbacks are exhausted: if at least one stage failed with a transport/API error the marker is kept (retry can still help — #114 semantics unchanged), but if every stage ran cleanly and returned empty the marker is deleted, the audio is kept in `history/`, and the console says so honestly — a new calm "no speech found in this recording — the audio is kept in history" panel (yellow, no retry hint) instead of the generic FAILED-plus-fix-your-key panel. This verdict is drawn only on the Soniox Live path, whose fallback chain reports whether any stage errored; a file-based engine (the Soniox upload slot or either Groq slot, reachable with `Ctrl+Alt+L`) swallows transport errors internally and returns empty either way, so its empty result can't be told apart from silence and stays the cautious FAILED + retryable — which is also what keeps a switch-and-retry a working way around an outage. One `Ctrl+Alt+R` on such a recording (retried on Soniox Live) now clears it for good, while a retry that fails for API/network reasons still persists across restarts and re-announces once (#134). The same honest verdict is applied to the original in-session attempt on that path, so a genuinely silent dictation on Soniox Live no longer writes a retry marker in the first place — removing #134's nag at its root. Off-Windows coverage of the marker lifecycle and of this live-vs-file verdict routing lands in the new `test_retry_marker_lifecycle.py`.

- A stalled microphone no longer freezes the whole app (#128). When an audio source stopped delivering mid-recording — the live incident was a Bluetooth headset whose capture stream wedged at driver level, but any stalled device qualifies — the blocking `stream.read()` never returned while holding the stream lock, so the very next stop/cancel/exit deadlocked behind that lock on the hotkey-listener thread; the recording loop died, every hotkey including `Ctrl+Alt+4` went dead, and the microphone stayed on until the process was force-killed. Three cooperating layers remove the freeze. **(1) The capture read can no longer block** — `record_chunk()` polls how much audio is buffered with the lock released between polls and only issues the read once a full chunk is ready (so it returns at once), keeping the same one-chunk-per-call contract and cadence with no perceptible added latency (the poll interval is ~5 ms, well under the ~64 ms chunk period); a **stall watchdog** treats no audio for `AUDIO_STALL_TIMEOUT_SECONDS` (5 s) as device loss and routes the session into the existing #49 device-loss endgame, so whatever was captured before the stall is saved and armed for `Ctrl+Alt+R` retry, exactly like a mid-recording device loss. **(2) No critical thread ever waits unbounded or runs a native audio close** — `_close_stream()` acquires the lock with a timeout and, on success, hands the actual `stop_stream()`/`close()` to a throwaway worker bounded by a join, because those calls block just as long as the read on a driver-wedged stream (the force-kill during the incident briefly reset the entire Bluetooth stack, proving the wedge sat at driver level); on either timeout the tool **abandons the wedged stream untouched** (leaks the handle; the OS reclaims it on exit) so hotkeys, the console, and the exit path stay alive. **(3) Reopening the device is bounded too** — the next recording's PyAudio init + stream open run on a worker with a timeout, since a start right after a driver wedge can hang the same way, and a fresh pipeline is built from scratch (the abandoned handles are dropped without `terminate()`, which would itself block on the wedged driver). Net effect: a stalled or dead microphone ends the current recording within a few seconds with the audio-so-far saved, the exit and all hotkeys keep responding throughout, and the next `Ctrl+Alt+W` reinitializes the device and records normally instead of the app hanging until a force-kill. New `AUDIO_READ_POLL_SECONDS` / `AUDIO_STALL_TIMEOUT_SECONDS` / `AUDIO_CLOSE_LOCK_TIMEOUT` / `AUDIO_OPEN_TIMEOUT_SECONDS` config constants (internal safety timers, no user surface); a new off-Windows regression test `test_audio_stall.py` fault-injects a wedging fake stream to prove the deadlock reproduction returns promptly, the watchdog fires, and every close/open path stays bounded.
- A failing V2 fallback no longer floods the console with a multi-line gRPC error repr (#124). #117 kept exception *tracebacks* off the console, but the empty-live fallback cascade's error line interpolated `str(e)` straight into the log **message** (`Fallback (Soniox V2 (sync)) raised for sequence N: {e}`), and a gRPC `_InactiveRpcError` renders `str()` as an ~8-line block (status / details / debug_error_string) — so with the network down the raw RPC internals landed red between the strips and buried the FAILED panel, the exact symptom #117 removed, arriving through a different mechanism (`prepare()` clears `exc_info`/`exc_text`/`stack_info` but never touches the already-interpolated message). A new `transcriber._one_line_error(e)` helper collapses any exception to one console-safe line — for a gRPC error the status code plus the first line of its `details()`, otherwise the class name plus the first non-empty line of `str()` — and the cascade error line (`thoughtborne.py`) plus the Soniox async REST error lines (`transcriber.py`) now pass `e` through it while keeping `exc_info=True`, so `thoughtborne.log` still records the full exception. Console-only fix; the httpx `HTTPStatusError` two-liner on the async path is covered by the same change.
- A failing transcription cascade no longer buries the console in raw tracebacks (#117). When a network outage made every engine fail in turn (Soniox Live/websockets → V2/gRPC → V4/httpx), each `logger.error(..., exc_info=True)` printed its full ~100-line Python traceback to the console, styled red, between the REC strip and the FAILED panel — the Cockpit's receding one-liner intent (#109) broke exactly when the user most needs orientation. Cause: console records travel through a `QueueHandler`, whose `prepare()` pre-formats every record with the handler's default formatter, which appends the exception text and bakes it into `record.msg` (the file handler runs first and caches the rendered traceback on `record.exc_text`), so `ConsoleFormatter` — which renders only a dim `HH:MM:SS  message` line and never appends exception text — merely styled the baked-in blob red. `DroppingQueueHandler.prepare()` now copies the record and clears its `exc_info`/`exc_text`/`stack_info` before queuing, so the console shows only the red one-liner plus the FAILED panel while `thoughtborne.log` keeps every full traceback via the independently-formatted file handler. Console-only fix: the file log's diagnostic depth is unchanged, and `raw_console` panels/strips (which carry no exception info) are untouched.
- Audio device names with non-ASCII characters no longer appear as mojibake in the log and console (#113). PortAudio (≥ v19.7) returns device names as UTF-8 bytes on Windows, but PyAudio decodes them with the ANSI code page (CP1252 on German Windows), so the UTF-8 bytes of "ö" reached Python already corrupted as the pair "Ã¶" — every German-named Bluetooth device ("Kopfhörer", "Freisprecheinrichtung", …) logged garbled. A new module-level helper `_repair_device_name()` re-encodes the name as CP1252 and decodes it as UTF-8 at the two places the default-input-device name is read (`_ensure_pyaudio_ready`, `_reinitialize_stream`); the round-trip is self-correcting, so a pure-ASCII or already-correct name — and a non-`str` value — passes through untouched and the repair can never double-apply. Display-layer fix only: device selection is by index, so recording is unaffected.
- The `Ctrl+Alt+Ü` self-test now names the actual Groq model instead of a generic "Groq" (#109, part of the #61 follow-ups). `GroqTranscriber.test_transcription` logged a hardcoded `Testing Groq with file: ...`, so the two free Groq slots (`whisper-large-v3` and `whisper-large-v3-turbo`) were indistinguishable in the self-test log. All four transcribers' self-test lines now derive the label from `get_name()` — the single display-label source — so the line reads `Testing Groq Whisper Large v3` / `Testing Groq Whisper Turbo v3` (the Soniox labels were already correct and are unchanged by the unification).
- A clean exit during an active recording now offers the retry on the next start (#106). Ending the program mid-recording — the exit hotkey `Ctrl+Alt+4`, or `Ctrl+C`/`Ctrl+Break` — already salvaged the in-flight audio to a bare `voice_<ts>.mp3` (#49 layer 1), but left nothing to re-arm `Ctrl+Alt+R`: the salvage discards the crash-safety `.partial` sidecar once the save succeeds and startup recovery only ever scanned for `.partial` files, so `Ctrl+Alt+R` reported "no failed transcription to retry" — the far likelier accidental clean exit (a mistyped hotkey, a long recording forgotten mid-thought) was handled worse than a hard kill. The salvage now leaves a small one-shot marker beside the archive, written *after* the archived audio exists but *before* the sidecar is discarded, so a marker that cannot be written keeps the sidecar and the existing kill-recovery still rescues the audio — the audio is never left un-recoverable. A second startup scan, in its own guard so a fault in one source cannot suppress the other, finds the markers, arms the single retry slot (#24) with the newest recording across both sources, and consumes each marker as it reads it — so the retry is offered exactly once across restarts, matching the existing kill-recovery.
- The filler filter no longer strips fillers the user deliberately quotes (#101, follow-up to #97). When a filler is spoken with quoting emphasis the STT models render it in straight double quotes (`"ähm"`); the #97 filter removed the word and left the quotes empty, collapsing `Er sagte "ähm".` to `Er sagte "".` — so a sentence *about* the word "ähm" was impossible to dictate. A filler whose match falls inside a balanced pair of straight double quotes is now left verbatim, quotes and all, and no filtering happens anywhere inside a quoted span, so even a longer quoted passage (`"... ähm ... äh ..."`) survives exactly as transcribed. Quote pairs are taken positionally (1st–2nd, 3rd–4th, …); a lone unpaired quote protects nothing, so a stray `"` can never switch the filter off for the rest of the transcript. Everything outside quotes filters exactly as before — proven bit-identical to the shipped behaviour on quote-free input — and because the exemption lives in the shared `_remove_spoken_fillers`, it applies on every engine at once.
- Raw library stderr output no longer appears twice on the console, and is no longer blanket-classified as WARNING (#39). The `StreamToLogger` stderr redirect both passed the text through to the real console and logged it at WARNING — which the console log handler then printed a second time, formatted. Redirected stdout/stderr now log to a file-only logger (`Thoughtborne.stdio`): the raw passthrough is the single console copy, and the file log still records every line — at INFO with an `[stderr] ` prefix for searchability instead of the blanket WARNING. Display-layer fix; no diagnostic line is dropped (known trade-off: a library crash traceback now files as INFO, while staying fully visible raw on the console).
- Device loss during a recording no longer leaves a zombie session (#49). When the microphone vanished for good (e.g. Bluetooth headset died) and stream reinitialization failed, `record_chunk()` silently set `recording = False` and nobody told the app layer: the recording loop idled, the next stop hotkey took the insert-last-text branch and typed the PREVIOUS transcript into the active window, and the dead session's frames — still intact in RAM — were wiped by the next `start_recording()`. The error path now pins the dying session's frames and sidecar writer and raises a `recording_aborted` flag that the recording loop consumes (guarded so a deliberate stop/cancel racing the error path costs at worst a duplicate archive file, never the audio); the loop hands the frames to a salvage worker that archives them and arms the retry slot. The practically unreachable "stream closed while recording" branch gets the same treatment instead of remaining a second silent zombie path.
- `get_audio_duration()` and `save_recording()` no longer depend on a live PyAudio instance (`self.p.get_sample_size` → module-static `pyaudio.get_sample_size`, value-identical). Latent crash: `self.p` is `None` before the first recording and potentially dead after a failed device reinitialize — exactly the states the new #49 salvage and recovery paths run in. Happy-path output stays byte-identical (verified against pre-change reference hashes).

First public release.

### Removed

- `soniox-v4` carousel slot — folded into the now-hybrid `soniox` slot (#31, see Changed). `AVAILABLE_APIS` is now `["soniox-live", "soniox", "groq-large", "groq"]` and the `Ctrl+Alt+L` carousel cycles four entries; `SonioxV4Transcriber` stays in the codebase as the slot's async engine (and the empty-live cascade's V4 stage) but is no longer user-facing: its `create_transcriber()` branch, the `[SONv4]` banner segment, and the `_get_api_emoji` branch are gone. No migration needed — the API selection was never persisted; a manually set `DEFAULT_API = "soniox-v4"` now surfaces at startup as a skip line quoting the factory's ValueError (listing the four supported names), after which startup falls through to the first constructible carousel entry (#40).
- Dropped the `modal` (Parakeet) and `huggingface` (Primeline Whisper) transcription APIs, trimming the runtime set from six to four (#3). `AVAILABLE_APIS` is now `["soniox-live", "soniox", "groq", "soniox-v4"]` and the `Ctrl+Alt+L` carousel cycles in that order; `DEFAULT_API` stays `soniox-live`. Removed the `ModalParakeetTranscriber` and `HuggingFaceTranscriber` classes, their `create_transcriber()` branches, the now-dead top-level `import requests` (used only by those two classes), the `HUGGINGFACE_*`/`MODAL_*` config settings and getenvs, the `modal_parakeet/` deployment folder, and the matching `.env.example` keys and README entries. The live Modal app `parakeet-german` is left running/idle (scales to zero) and its deploy code remains in git history; this change only unwires the API from the app.

### Changed

- The `soniox` slot is now a V2/V4 hybrid (#31). Recordings under `SHORT_AUDIO_THRESHOLD` (58 s) run V2 sync exactly as before, but when V2 raises (unreachable, retired, transport error) the slot transparently falls back to V4 async REST — one `[FALLBACK]` WARNING naming the gRPC status on the console, full trace at DEBUG (file-only, #32 pattern). Recordings of 58 s or more go straight to V4 async REST; the V2 gRPC async polling path is deleted. Two deliberate exceptions to the fallback rule: auth errors don't hop (V4 uses the same `SONIOX_API_KEY`, so a hop would only produce a second 401 and a duplicate `[AUTH]` line — the single `[AUTH]` line + empty result stay as before), and an empty V2 result without an exception (usually silence) doesn't hop either — only the empty-live cascade keeps its empty → V4 fall-through, where a dead live path already raised suspicion. A missing Soniox SDK now degrades instead of killing the slot: `SonioxTranscriber` no longer raises `ImportError` at init but logs one WARNING and serves every recording via V4 — without this, folding the `soniox-v4` slot away would have left pip installs without the optional SDK with no Soniox upload path at all. The empty-live cascade now calls the raw V2 primitive (`_transcribe_v2_sync`) so its V2 stage cannot double-fallback internally, and a V2 auth failure inside the cascade logs at DEBUG, leaving the following V4 stage's `[AUTH]` line standing alone (one line instead of today's two). The long-path switch is backed by a 13-sample / 41.9-min side-by-side quality gate of V2-gRPC-async vs. V4-REST on real dictation audio (aggregates: no V2-style end artifacts on V4 in 0/13 samples, sentence-final punctuation 13/13 vs. 12/13, more stable casing, slightly better proper nouns via context support, consistently faster on long files; but 173 verbatim inline fillers vs. 0 on V2 — addressed by the new V4 filler filter, see Added — and, as a known trade-off that is deliberately left unfiltered, V4 sometimes writes spoken number/version phrases out as words where V2 formatted digits, observed in 2/13 samples: too thin a pattern to act on). Full gate results live in the local `_research/2026-06_soniox-v2async-vs-v4-quality/`.
- Brand spelling unified: "GROQ" → "Groq" in prose, display names, the startup banner, log messages, and code comments, matching how the company spells itself (#38). The carousel entries now report "Groq (schnell)" / "Groq Large (genauer)" and the banner/status tags read `[Groq]` / `[Groq-L]`. Env var names (`GROQ_API_KEY`), Python identifiers (`GROQ_MODEL`, `GROQ_LARGE_MODEL`, `_clean_groq_hallucinations`), the lowercase carousel keys (`groq`, `groq-large`), and the hallucination-filter data are unchanged — no `.env` or config edits required, no functional change.
- `CLAUDE.md` now points to `VISION.md` as the calibration point for judgment calls, and the former "maintenance mode" rule is reworded: it still caps code churn (no refactoring/rewrites without a stated reason) but no longer reads as a feature freeze — direction and ambition live in `VISION.md` (#35).
- Live audio sample rate dropped from 44.1 kHz to 16 kHz mono (`config.RATE = 16000`, #2). Soniox V4 operates internally at 16 kHz and downsamples everything above it server-side; all six transcription APIs (Soniox V2/V4-async/Live, Modal Parakeet, HuggingFace Whisper, Groq Whisper) work natively at 16 kHz, so this is pure bandwidth saving with no recognition-quality loss. Live upload drops from ~705 kbps to ~256 kbps (-64 %), the root-cause counterpart to the Block-2 sender-thread fix that treated TCP-backpressure symptoms only. New MP3s in `voice_archive/` are 16 kHz mono; existing recordings stay 44.1 kHz. Diagnostic threshold for slow PyAudio `stream.read()` raised from 50 ms to 100 ms to match the new nominal chunk time (~64 ms at CHUNK=1024 / RATE=16000); the loop-iteration-gap threshold stays at 50 ms because it measures wallclock outside the read and is sample-rate-independent.
- `SonioxLiveTranscriber` now uses a producer/consumer architecture for the WebSocket send path. `send_audio_chunk()` is non-blocking: it enqueues the chunk on an internal `queue.Queue` (size `SONIOX_LIVE_QUEUE_MAX_CHUNKS`, default 50 ≈ 1.16 s) and a dedicated sender thread drains the queue to the WebSocket. TCP backpressure now stalls only the sender thread, so the recording loop keeps reading PyAudio without gaps even when sends block. Under heavy backpressure beyond the queue capacity, newest chunks are dropped (WARNING on drop-mode entry, INFO on recovery, per-chunk drops at DEBUG); the MP3 archive is unaffected because frames are stored in `audio_handler` independently of the send queue. Finalize uses a sentinel-based queue drain with a timeout fallback so it can't hang on a dead sender. Three new config constants (`SONIOX_LIVE_QUEUE_MAX_CHUNKS`, `SONIOX_LIVE_SENDER_JOIN_TIMEOUT`, `SONIOX_LIVE_FINALIZE_DRAIN_TIMEOUT`).

### Fixed

- The `Ctrl+Alt+L` carousel no longer gets stuck on an entry whose API key is missing, and startup no longer dies when the default API lacks its key (#40). `switch_api` previously advanced exactly one slot and stayed put on construction failure — so every further press retried the same failing neighbor (post-#36, a Groq-only user could switch from `groq-large` to `groq` once and then never get back). It now skips past non-constructible entries with one visible console line per skip (`Skipped soniox-live (SONIOX_API_KEY missing)`, WARNING over the #11 queue path) until an entry works; if the loop comes full circle it stays on the current transcriber and names the missing keys and that they belong in `.env`. Startup gained the same fallthrough: when `DEFAULT_API` cannot be constructed, the remaining carousel entries are tried in order and the chosen API is stated plainly in the log and the startup banner (`SONIOX_API_KEY missing -> started on groq-large (default: soniox-live)`); a `DEFAULT_API` edited to an unknown name now falls through with a skip line naming it instead of failing hard. When *no* entry is constructible — the newcomer-without-keys case — startup prints an actionable error block (which keys are missing and for which entries, that they go into `.env`, pointer to `.env.example`/README) instead of the previous traceback-plus-generic-init-error, then exits with code 1 after Enter; it deliberately does not keep running keyless, because keys are read once at import and a restart after editing `.env` is required either way. Missing keys are detected via a new `MissingAPIKeyError(ValueError)` carrying the env-var name, raised by all four transcriber constructors (messages unchanged); the constructors' `... not found in environment variables!` ERROR lines drop to DEBUG so the skip WARNING stands alone. Any other construction failure is skipped too, with `ExceptionType: message` as the skip reason. With all keys present, behavior is unchanged — startup and every switch succeed on the first construction attempt as before; the only console difference is that `Switching API from X to Y` now prints after the successful construction instead of before it, so the construction INFO lines precede it.
- The empty-live fallback cascade no longer logs an ERROR with a full stack trace when its V2 stage hits the expected SDK-less state (#40; surfaced by the #31 review). `_try_fallback` skips the V2 attempt with a single INFO line (`V2 fallback stage skipped for sequence N (Soniox SDK not installed)`) instead of calling `_transcribe_v2_sync` and catching its `RuntimeError`; the result is unchanged (empty string, the cascade falls through to V4 as before) — only the presentation changes. The one-time SDK WARNING at `SonioxTranscriber` construction stays.
- `[CLIPDIAG verdict]` no longer logs a misleading `PASTE DID NOT LAND` for targets outside the trusted text-control allowlist (#33). Live data from the #29 fix showed dictations into WezTerm, browsers and other custom-rendered controls produced `DID NOT LAND` verdict lines even though the paste had landed — `WM_GETTEXTLENGTH` on those classes falls through to `DefWindowProc`, which answers with the (constant) window-title length, so the before/after delta carries no information about the paste. The retry guard already excluded these classes via the `_TEXT_CONTROL_CLASSES` allowlist, so behaviour was correct; only the verdict log line lied and would have misdirected future forensics. The verdict block in `_insert_text_via_clipboard` now branches on the same allowlist: for trusted classes (Scintilla / Edit / RichEdit*) the `PASTE LANDED` / `PASTE DID NOT LAND` verdict and WARNING-on-miss stay exactly as before; for untrusted classes the line records `INCONCLUSIVE (untrusted length source)` at DEBUG with the raw `len_before`/`len_after`/`delta` numbers still logged. Pure logging-honesty fix; no behavioural change.
- Clipboard insertion no longer needs several `Ctrl+Alt+D` presses in Notepad++ when an image or a copied file was previously on the clipboard (#29). Root cause — proven live by a grayed paste icon in the failing state, an alt-tab heal, and a `GetMenuState` probe showing the stale state in both directions: Notepad++ caches its paste-enabled state (`checkClipboard` → `SCI_CANPASTE` → `enableCommand(IDM_EDIT_PASTE)`), registers no clipboard listener, and Ctrl+V is an accelerator on that command — Win32 swallows accelerators whose menu command is disabled, so after non-text content Thoughtborne's text copy went unnoticed and the injected Ctrl+V was consumed before ever reaching the editor (upstream as notepad-plus-plus#16456; the v8.8 fix covers only N++-internal copy/cut). Two-layer fix in `_insert_text_via_clipboard`: (1) when the prior clipboard content was non-text and the focused child window is a Scintilla control, send `SCI_SETXOFFSET` with the unchanged current offset before pasting — N++'s own `WM_ACTIVATE` heal path (`SCN_UPDATEUI` → `checkClipboard`), which re-enables paste now that the transcript is on the clipboard; (2) a verified single retry that fires only when `WM_GETTEXTLENGTH` of the focused control proves the paste did not land (honest-responder class allowlist, 250 ms re-measure, focus-unchanged check, before the clipboard restore) — so it cannot blind-double-paste. Targets that are not Windows text controls (browsers, terminals) keep the previous behaviour exactly. Full diagnosis with sources in `_research/2026-06_npp-paste-gate-clipboard/`.
- A rejected API key now leaves the uniform `[AUTH]` line standing alone on the console (#32). #10 introduced the line, but each auth branch also logged the full exception at ERROR level right after it — and the console handler passes ERROR — so the stack trace printed directly underneath and buried the actionable message. The auth branches (Groq `AuthenticationError`, Soniox V2 gRPC `UNAUTHENTICATED`, Soniox V4 HTTP 401, Soniox Live in-band `error_code` 401) now log the trace/detail at DEBUG, which is file-only; `thoughtborne.log` keeps the full stack trace as before. Non-auth errors keep their ERROR-level console behaviour. On the Soniox Live path the demotion covers the receiver's 401 detail line, the finalization's `session had error` repeat, and the sender thread's send-error line for the case where the server closes the WebSocket right after a 401 — the latter two keyed to a new `_session_auth_error` flag that rides the same `_result_ready` synchronization as `_session_error`.
- Clipboard insertion (`Ctrl+Alt+D` / `Ctrl+Alt+H`) no longer fails when the clipboard holds non-text content — a copied image, a copied file, or any non-CF_UNICODETEXT format (#23). pyperclip's Windows backend is text-only, so `pyperclip.paste()` returns `""` for such content (indistinguishable from an empty clipboard, not an exception). The restore step then ran `pyperclip.copy("")` ~100 ms after `Ctrl+V` (`CLIPBOARD_RESTORE_DELAY`), wiping the clipboard before the target app had finished pasting — so the transcript often never landed. `_insert_text_via_clipboard` now restores the prior clipboard only when a non-empty text string was actually saved (`if original_clipboard:` instead of `is not None`); for non-text or empty prior content it skips the restore entirely and the transcript stays on the clipboard. Prior **text** is still restored unchanged. No new dependency and no clipboard-format introspection — preserving the original image/file is explicitly out of scope.
- Soniox Live self-test (`Ctrl+Alt+Ü`) now streams the test file at its own sample rate instead of `config.RATE` (#12). `SonioxLiveTranscriber.test_transcription` read `test_audio.mp3` at its real 44.1 kHz but announced `config.RATE` to the Soniox session and paced at it; after #2 dropped `RATE` to 16 kHz the two diverged, so Soniox decoded the bytes ~2.75× too slow and returned an empty transcript. `start_session()` gained an optional `sample_rate_override` (default `config.RATE`, so the production live-capture path is unchanged) that the test now sets to the file's actual rate, with pacing switched to `CHUNK / samplerate`. The self-test passes again regardless of `config.RATE`.
- Invalid or expired API keys now surface one uniform, actionable console line on every transcriber instead of a cryptic generic error or, in one case, a silent hang (#10). A shared `AbstractTranscriber._report_auth_failure(env_var, detail)` helper logs `[AUTH] <name>: API key rejected (<detail>). Check <ENV_VAR> in .env, then restart Thoughtborne.` at ERROR — the console-visible, non-blocking path established by #11 — while the full stack trace stays in `thoughtborne.log`. Each transcriber detects its own auth signature: Groq `AuthenticationError` (invalid vs `expired_api_key` distinguished via `str(e)`, because `body['code']` is `None` in groq 0.29), Soniox V4 `httpx.HTTPStatusError` 401, Soniox V2 gRPC `StatusCode.UNAUTHENTICATED`, Soniox Live `error_code == 401` in the receiver-thread frame, and HuggingFace HTTP 401. The Soniox V2 async polling loop now re-raises an unauthenticated gRPC error instead of swallowing it per attempt, so a bad key on the long-recording path fails fast rather than looping for the full ~300 s timeout. Modal Parakeet is excluded (no API key — the endpoint URL is the secret).
- `HotkeyManager.stop()` no longer self-joins the listener thread on the `Ctrl+Alt+4` exit path (#14). The exit hotkey's callback runs *on* the listener thread, so `stop_program → stop()`'s `self._thread.join()` raised `RuntimeError: cannot join current thread`. The error short-circuited `stop_program`'s own teardown, so `output_manager.stop()` and the active-processing-thread joins never ran on that path -- an in-flight transcription was not awaited on exit (cleanup still happened redundantly via `run()`'s `finally`, so the program exited, just uncleanly). `stop()` now detects `threading.current_thread() is self._thread`, posts `WM_QUIT` to unwind the message pump, and returns without joining; `run()`'s `finally` calls `stop()` again from the main thread to do the real join. Exit is clean (no `RuntimeError` in the log) and `stop_program`'s teardown runs.
- Listener-thread hang under a Windows-cmd Mark/Quick-Edit selection (#11). Every `logger.info/warning/error` call from the hotkey-listener thread previously blocked synchronously on the cmd console when the user had selected text with the mouse, because the console `StreamHandler` wrote directly to the real `sys.stderr` and `emit()` writes are synchronous. A four-second selection during start-recording reproduced the freeze; the original 28-second incident was the same mechanism held longer. Console logging now goes through a `QueueHandler` + `QueueListener` (Python `logging.handlers` standard pattern): the listener thread enqueues records non-blockingly and a dedicated daemon thread drains the queue to stderr. When cmd blocks the drain, only that daemon stalls -- the hotkey listener stays responsive and the file log is unaffected (it runs through a separate synchronous `RotatingFileHandler` that never touches stdout/stderr). All `print()` calls in the listener-thread callbacks (`on_start_recording`, the four `on_stop_recording_*`, `on_cancel_recording`, `on_exit_program`/`stop_program`, the `on_switch_api` and `on_test_transcription` paths, `_handle_mistrigger_during_recording`, `print_ready_status`, plus `start_processing_thread` -- which is not itself a callback but runs synchronously in the listener thread because all four `on_stop_recording_*` callbacks call it -- and the device-changed `print` in `audio_handler._ensure_pyaudio_ready`) were replaced with `logger.info/warning/error` so they ride the same non-blocking path -- otherwise `print` would still block via the `StreamToLogger` redirect on `sys.stdout`. Redundant `[HH:MM:SS]` status lines that duplicated an adjacent log line were dropped (LOG_FORMAT already carries the timestamp). New `LOG_CONSOLE_QUEUE_MAX` config (default 200) bounds the queue; when cmd holds the drain longer than the queue can absorb, the newest console records are dropped (the file log keeps the complete record either way). The drop goes through a `DroppingQueueHandler` subclass whose `enqueue` swallows `queue.Full`: the stock `QueueHandler` would instead route the overflow into `handleError`, which writes a traceback to the same blockable console stderr and would re-hang the listener on the 201st record -- so the overflow path is non-blocking too, not just the steady state. The shutdown path is likewise guarded: `cleanup()` wraps `QueueListener.stop()` in `try/except` because `stop()` enqueues its sentinel via `put_nowait` and would raise `queue.Full` (crashing cleanup) if cmd has filled the queue at exit; swallowing it is safe since the daemon listener dies with the process and the file log is already complete. The Pass-1 DEBUG markers from `cfc3645` stay in place as forensics for the residual hypotheses (PyAudio reinit stall, race in `start_recording`) that Pass 1 did not exclude but did not observe either.
- `SonioxLiveTranscriber.transcribe` now calls `_close_session_internal()` on the early-return path ("No active Soniox Live session to finalize") as well, so per-session send-latency and queue-drop stats are logged even when finalize hits a session that already died (e.g. from a 20-s Soniox idle timeout during prolonged TCP backpressure). Previously those exact sessions — the most diagnostically interesting ones — silently skipped the stats line.
- `SonioxLiveTranscriber._receiver_loop` now resets `_session_active` to `False` in its `finally` block. Without this, every audio chunk sent after a server-initiated WebSocket close produced a fresh ERROR log entry until the user pressed Stop (observed: 2000+ spam entries within seconds of a single disconnect).

- `Ctrl+Alt+R` retries the most recent **failed** transcription (#24). When a recording's transcription fails inside the software -- empty after the live→V2→V4 fallback chain, or a worker exception -- the app arms a single in-memory slot with the archived `voice_archive/voice_<ts>.mp3` path and its duration. Pressing `Ctrl+Alt+R` spawns a normal transcription job (new sequence number) that re-runs the archived MP3 through the existing file-fallback chain (`_run_empty_transcript_fallback`, duration-gated Soniox V2/V4) and inserts the result at the cursor like a fresh dictation -- never blocking or reordering the queue. A **user cancel** (`Ctrl+Alt+X`) never becomes a retry target: a cancelled recording never reaches a worker, so the slot is only ever written from inside the worker path. Later *successful* transcriptions don't clear the slot; only a *successful retry* resolves it (via an identity check, so a newer failure that lands mid-retry isn't clobbered), and a failed retry stays retryable indefinitely. Only the single most-recent failure is recoverable; the slot is in-memory only (lost on restart). The retry insert mirrors the keyboard stop hotkey's `wait_for_key_release` so the still-held `Ctrl+Alt+R` modifiers don't misfire the typed insertion. Binding lives in `config.HOTKEYS` for easy relocation.

- Empty-transcript fallback for Soniox Live (#1, Block 3). When `SonioxLiveTranscriber.transcribe()` returns an empty string -- typically a Class-B failure where the server closed the WebSocket (`1011 keepalive ping timeout`) before the stop hotkey under sustained TCP backpressure -- `process_recording_thread` now retries the just-saved MP3 (same content as the archived copy in `voice_archive/`) against a file-based Soniox API instead of marking the recording as failed. Choice depends on duration: recordings shorter than `SHORT_AUDIO_THRESHOLD` (58 s) go to `SonioxTranscriber` (V2 sync, ~2-3 s for 30 s audio) first, with a fall-through to `SonioxV4Transcriber` if V2 raises or returns empty -- so a future V2 shutdown by Soniox leaves the fallback chain working. Recordings of 58 s or longer go straight to `SonioxV4Transcriber` (V4 async polling, ~10-60 s; the only option past V2's 60 s hard limit). Both fallback transcribers are lazily instantiated as singletons on first use, guarded by a small init lock so parallel Class-B disconnects don't race on construction. A clearly framed console block plus INFO log entry surface the trigger; an additional INFO line plus console line marks the V2-to-V4 fall-through when it happens. If every available fallback returns empty / fails, the exhausted-stages line is logged at ERROR and the existing is_error path takes over. Trigger is restricted to `SonioxLiveTranscriber` -- empty results from V2 / V4-async / Modal / HuggingFace / Groq already mean the file-based path was tried and failed, so a second pass would not help. The fallback is not interruptible (Ctrl+Alt+X is a no-op once recording has stopped).
